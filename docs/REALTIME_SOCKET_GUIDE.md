# 实时通信与房间同步说明

## 为什么使用 Socket.IO

这个项目不是一次性提交表单的网页，而是一个多人同时在线协作的 BP 系统。

它需要解决的核心问题是：

- 房主操作后，其他人要立刻看到结果
- 参与者预选英雄时，房间内其他人要同步看到预览
- 房间暂停、恢复、删除时，所有客户端都要立即更新
- 大厅中的房间状态也要跟着变化

因此这里采用：

- REST 处理初始化请求
- Socket.IO 处理实时同步

## 当前使用到的核心事件

### 大厅相关

- `lobby:list`
  获取当前大厅房间列表。
- `lobby:rooms`
  服务端主动广播最新大厅列表。

### 房间订阅相关

- `room:watch`
  输入正确密码后，开始订阅房间状态。
- `room:join`
  正式以参与者或观众身份加入房间。

### 房主管理相关

- `room:start`
  房主开始 BP。
- `room:swap-sides`
  房主交换双方阵营。
- `room:pause`
  房主暂停当前 BP。
- `room:resume`
  房主恢复当前 BP。
- `room:undo`
  房主返回上一步，撤回最近一次确认的 ban/pick。
- `room:reset`
  房主重置房间，开始新的对局。
- `room:delete`
  房主删除房间。

### BP 相关

- `draft:preview`
  同步当前回合的预选英雄。
- `draft:preview-state`
  广播房间内双方当前的预选状态。
- `draft:action`
  提交一手正式的 ban 或 pick。

### 状态广播

- `room:state`
  广播最新房间状态。
- `room:error`
  广播房间级错误提示。
- `room:deleted`
  房间被删除后通知所有房间内用户退出当前房间。

## 一个完整回合是怎么同步的

### 预选阶段

当前行动方点击英雄头像后，前端不会立即提交正式操作，而是先发出 `draft:preview`。

服务端记录这次预选后，会把最新预选状态广播给房间内所有人。这样参与者、房主和观众都能看到当前预览。

### 确认阶段

参与者点击确认后，前端发出 `draft:action`。服务端会校验：

- 当前用户是否是参与者
- 当前是否轮到该阵营
- 操作类型是否正确
- 英雄是否已经被选过

校验通过后，服务端写入 `DraftAction`，推进回合，并广播新的 `room:state`。

## 返回上一步是如何工作的

房主触发 `room:undo` 后，服务端会：

1. 找到当前对局中最后一条 `DraftAction`
2. 删除这条记录
3. 把当前回合索引退回一格
4. 重新广播最新房间状态

这意味着撤回不是前端假效果，而是真正修改数据库里的最近一步。

## 暂停与恢复是如何工作的

暂停时，服务端记录 `pausedAt`。

恢复时，服务端会根据暂停持续时间，把当前回合开始时间向后平移。这样倒计时不会因为暂停而白白流逝。

## 本地身份恢复

为了让刷新页面后还能识别“我是谁”，前端会在 `localStorage` 里存两类信息：

- 当前房间的密码
- 当前用户在该房间内的成员身份

页面刷新后，前端会重新拉取房间状态，并尝试从房间成员列表中匹配本地身份。匹配成功后，就能恢复成房主、参与者或观众视角。

## 测试建议

本地测试多身份时，推荐这样做：

- 一个普通浏览器窗口作为房主
- 一个无痕窗口作为参与者
- 另一个浏览器或另一个用户配置文件作为另一位参与者或观众

这样可以避免多个窗口共享同一份本地存储，导致身份混在一起。
